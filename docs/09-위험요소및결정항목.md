# 09. 위험 요소 및 결정 항목

## 9.1 기술적 리스크

### 9.1.1 Wi-Fi 신호의 변동성

**현상**

Wi-Fi RSSI는 동일 위치에서도 다음 요인에 의해 ±5~10 dBm의 변동을 보일 수 있다.

- 사람의 통행 (인체에 의한 신호 흡수)
- 시간대별 AP 부하
- 디바이스 자세
- 다른 무선 장비의 간섭

**영향**

위치 추정 정확도 저하. 다른 노드로 오인할 가능성.

**대응 방안**

| 우선순위 | 대응 | 비고 |
|---|---|---|
| P1 | 노드별 다회 측정 (10회 이상)으로 평균 패턴 학습 | 데이터 수집 단계에서 적용 |
| P2 | KNN의 K값을 3~5로 설정해 다수결 안정화 | §4.3.3 참조 |
| P3 | (선택) 3초간 다중 측정 후 다수결 | 시간 여유 시 도입 |

### 9.1.2 KNN 정확도 한계

**현상**

샘플이 부족한 노드, 또는 신호 패턴이 인접 노드와 유사한 경우 분류 정확도가 떨어진다.

**대응 방안**

- 시연 노선·역을 1개로 한정하여 데이터 수집 부담을 낮추고 품질을 확보
- 인접 노드의 신호 패턴이 너무 유사하면 노드 위치 자체를 조정 (지하철역 답사 단계에서 결정)
- 사용자에게 "위치 인식 실패 시 재요청" 음성 안내 옵션 추가

### 9.1.3 데이터 수집의 부담

**현상**

각 노드에서 10회 이상 측정 → 노드 30개 가정 시 최소 300회 측정. 시간과 인력 소요가 큼.

**대응 방안**

- 팀 전원이 1회 답사 시 노드를 분담해 동시 수집
- 측정 자동화 미니 앱 개발 검토 (한 번에 여러 회 측정 후 CSV export)
- 시연 범위를 **1개 역사의 일부 구역**으로 한정

### 9.1.4 나침반 센서의 부정확성

**현상**

스마트폰 자기센서는 주변 금속 구조물·전자기 간섭에 민감. 지하철역(철골 구조)에서는 오차가 클 수 있다.

**대응 방안**

- 진동 안내의 임계 각도를 **±15°** 까지 허용 (요구사항 그대로 준수)
- 사용자에게 진입 시 한 차례 "보정 자세" 안내 (정북을 향하게 후 시작) — 앱 측 구현
- 시연 환경에서 사전 테스트로 오차가 큰 위치를 회피

### 9.1.5 통신 실패

**현상**

지하철역의 Wi-Fi 환경 또는 모바일 데이터 연결이 불안정할 수 있다.

**대응 방안**

- 시연 환경은 안정된 네트워크 사전 확보
- 앱 측 재시도 정책 정의 *(09.3 결정 항목 참조)*
- 통신 실패 시 사용자 음성 안내 ("일시적인 연결 오류입니다")

## 9.2 일정 리스크

### 9.2.1 PPT 일정과 변경된 시스템의 정합성

**현상**

4월 PPT 발표 시점 일정은 ESP32 비콘·React Native 기준이었으나, 시스템이 Wi-Fi + 안드로이드 네이티브로 변경됨.

**영향**

- 6~7주차의 ESP32 회로 작업 → **무효** (실제로는 Flask 골격 작업으로 대체)
- 8~9주차의 React Native 앱 작업 → **안드로이드 네이티브 앱** 으로 대체
- 마일스톤 날짜는 유지 가능하나, **각 주차 실작업이 PPT와 다름**을 명시 필요

**대응 방안**

- 본 보고서 **§8.2.1 표** 에서 "새로운 구성에 맞춰 작업 항목 재해석" 명기
- 중간 미팅(5/15) 시 교수님께 시스템 변경 이력 명확히 공유

### 9.2.2 안드로이드 학습 곡선

**현상**

팀원 C·D가 안드로이드 네이티브 개발에 익숙하지 않을 경우 9주차 일정에 차질 발생 가능.

**대응 방안**

- 9주차 통신 테스트는 **간단한 더미 앱** (버튼 한 개로 API 호출 후 화면 표시)으로 시작
- TTS·진동·Wi-Fi 스캔 기능은 단계적으로 추가
- 정 안되면 시연용으로 **Postman/curl 시나리오** 백업 준비

### 9.2.3 데이터 수집의 일정 영향

**현상**

데이터 수집은 주로 12~13주차에 배치되어 있으나, 시연 노드 수에 따라 시간이 더 걸릴 수 있다.

**대응 방안**

- 11주차 중간 미팅 직후 **즉시 1차 데이터 수집** 착수
- 노드 수를 최소화 (10개 내외) 하여 시연 가능 범위 확보

## 9.4 종합 대응 전략

### 9.4.1 우선순위 원칙

> "동작이 우선, 정확도는 그 다음"

| 단계 | 목표 |
|---|---|
| 1단계 (~10주차) | 시연 시나리오 1개를 **끝까지 동작**시키기 (성능 무관) |
| 2단계 (11~13주차) | 데이터 수집 → 위치 추정 정확도 개선 |
| 3단계 (14주차~) | 시연 보정 → 발표 준비 |

### 9.4.2 리스크 모니터링

| 주차 | 점검 항목 |
|---|---|
| 7주차 종료 시 | Flask·DB 골격이 동작하는가? |
| 9주차 종료 시 | 더미 데이터로 3개 API가 모두 응답하는가? |
| 10주차 종료 시 | 앱→서버 통신이 한 번이라도 성공하는가? |
| 11주차 (중간 미팅) | 교수님께 시연할 수 있는 흐름이 있는가? |
| 13주차 종료 시 | 실제 데이터로 시연 시나리오가 완주되는가? |

각 점검 시점에서 미달 시 **단순화 옵션 추가 적용** 또는 **시연 범위 축소**로 대응한다.

### 9.4.3 보고서 작성 시 활용

본 문서의 리스크 분석 결과는 **중간 미팅 발표 자료** 및 **최종 보고서의 위험 관리 섹션**에 그대로 활용 가능하도록 작성되었다.
